自建 Tailscale DERP 中继实践

发表于 2026-05-03 分类于 学习印记 阅读次数：
记录在只有公网 IP、没有域名的云服务器上搭建 Tailscale 自建 DERP 中继的完整过程，包括 DERP 原理、证书方案选择、systemd 部署和踩坑复盘。

最近使用 Tailscale 连接多台设备时，发现有些网络环境下直连并不总是稳定。虽然 Tailscale 的体验已经非常接近“无感组网”，但当两端网络都比较复杂时，连接还是可能会退回 DERP 中继。默认的 DERP 节点大多在香港、新加坡、日本等地，对我来说有时延迟还可以，有时又不够理想。

手里刚好有一台国内云服务器，有公网 IP，而且长期在线。于是很自然地产生了一个想法：能不能把这台服务器配置成自己的 Tailscale DERP 中继，让 Tailscale 在需要中继时优先走一个更近、更可控的节点？

这篇文章记录这次配置过程。为了避免泄露敏感信息，文中的公网 IP、证书指纹、密钥等都使用占位符表示。

DERP 是什么

Tailscale 的底层是 WireGuard。正常情况下，两台设备之间会尽量通过 NAT 穿透建立直连。直连成功之后，数据包直接在两端之间传输，延迟最低，吞吐也最好。

但是现实网络并不总是理想的。比如：

  1. 两端都在比较严格的 NAT 后面。
  2. 某些网络限制 UDP。
  3. 校园网、公司网、运营商网络策略比较复杂。
  4. 两端公网路径质量不稳定。

这时 Tailscale 就会使用 DERP 作为兜底通道。DERP 可以理解成一个中继服务器，负责转发已经端到端加密的 WireGuard 流量。DERP 本身并不能解密通信内容，它只是保证在直连失败时，两台设备仍然能够连上。

所以自建 DERP 的意义不是替代 WireGuard 直连，而是提供一个更近的兜底节点：

  1. 降低中继路径延迟。
  2. 避免流量绕到较远地区。
  3. 提升特殊网络环境下的可用性。
  4. 对中继节点的位置和运行环境有更多控制权。

初始条件

这次使用的是一台 Ubuntu 22.04 云服务器，只有公网 IP，没有绑定域名。服务器上原本已经有一些 Docker 服务和代理服务在运行，但 DERP 最终没有使用 Docker 部署，而是直接作为 systemd 管理的系统进程运行。

Tailscale 官方对自建 DERP 的要求主要包括：

  1. 服务器需要公网可达。
  2. TCP 443 用于 DERP HTTPS。
  3. TCP 80 用于 HTTP 辅助服务。
  4. UDP 3478 用于 STUN。
  5. DERP 服务需要 TLS。

这里最大的限制是：没有域名，只有公网 IP。

按照普通 HTTPS 服务的思路，如果没有域名，就很难从 Let’s Encrypt 申请常规证书。最开始我也尝试过用通配 DNS 服务绕过这个问题，但最后发现 Tailscale 官方其实已经内置了更适合纯 IP 场景的方案。

关于重启和可用性

因为 derper 是通过 systemd 管理，并且已经执行了：

所以服务器重启后，DERP 服务会自动恢复。

重启期间仍然会有短暂影响：

  1. 如果没有设置 OmitDefaultRegions，Tailscale 可以回退到官方 DERP，连接一般仍然可用，只是路径可能绕远。
  2. 如果设置了 "OmitDefaultRegions": true，则服务器重启期间，自建 DERP 不可用，某些不能直连的设备可能会暂时失联。

所以更稳妥的方式是先不要急着禁用官方 DERP。等自建 DERP 稳定运行一段时间后，再决定是否要完全只使用自己的 DERP 节点。

小结

这次配置最大的收获是重新理解了 Tailscale 的连接路径：直连仍然是最优解，DERP 是兜底；自建 DERP 的价值不在于让所有流量都经过自己的服务器，而是在直连失败时提供一个更近、更稳定的中继点。

对于只有公网 IP、没有域名的服务器，最终采用的方案是：

  1. 使用官方 derper。
  2. 通过 systemd 作为系统服务运行。
  3. 使用 --certmode=manual --hostname=<SERVER_IP> 自动生成 IP 自签证书。
  4. 在 DERPMap 中使用 sha256-raw 证书指纹 pinning。
  5. 显式配置 IPv4 和 IPv6: "none"。

最终效果是，Tailscale 网络中新增了一个自定义 DERP region。对于网络环境复杂、直连不稳定的设备，它可以作为一个更近、更可控的兜底路径。
